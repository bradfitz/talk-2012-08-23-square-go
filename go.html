Go: why?

Current languages are too at the edges of a spectrum:

Fast code, but lots of it, verbose, repetitive, tedious, slow to build.Fast code, fun & easy to write, safeSlow code, fun & easy to write, explode at runtime, do one thing at a time
C++, Java, ...????Perl, Python, Ruby, JavaScript, PHP, ...

Go ...

is statically typed,

... but has type-inference

generates fast code, on par with C or C++

... but feels like Py/Ruby/JS/etc (closures, brevity, etc)

compiles super quickly (A -> B -> C needs only consult B when compiling A)

... enforces perfect dependencies (IWYU safe)

(continued)

Go ...

gives you control of memory layout (yay speed)

has pointers

... but no pointer arithmetic, no stale pointers ('stack', 'heap')

is safe

is garbage-collected

.. but lets you avoid tons of garbage (unlike Java)

.. allows structs in structs, pass structs by value, not just reference

Go...

kills the threads vs. events debate

... you can write sequential, readable code again! die, callback flow control, die.

has exceptions ("panic", "recover")

... but not exceptional flow control (no "try", "catch")

(continued)

Go ...

lets you write fun, high-level code to begin with

... but you make it fast later if it matters

... twiddle bits, layout memory, etc.

... but it's still readable!

... decades of C cruft, gone.

... without huge calling convention overhead (Perl/XS, SWIG, etc)

(continued)

Favorite feature...

synchronous APIs; no threads vs. events/callbacks

No dual set of APIs (sync vs async)

No callbacks required (can if you want)

Concurrency is built-in to the language, so APIs don't need to provide it

Block away! Who cares. Runtime will epoll you or thread you as needed.

Running? Get a thread. In a syscall? Get a thread.

Blocked on IO? epoll_wait. (or OS-equivalent)

Blocked on mutex or channel send/receive? runtime scheduler's waitlist.

All automatic

"runtime" is not a separate JVM-like thing. Compiled in to your static binary.
